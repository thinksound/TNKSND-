Active Noise Cancellation (ANC)

An interactive web app that senses ambient noise with a mic and emits "anti-noise" through a speaker to destructively cancel it — using the FxLMS adaptive algorithm.

Purpose of this document: explain how ANC works and how we propose to build the app, for your review before coding. Decision points are flagged in yellow boxes and summarized at the end.
⚠ Read this first — the honesty section. Real acoustic ANC has a hard causality/latency requirement: the anti-noise must reach the cancellation point at the same instant as the noise, phase-inverted. That demands microsecond-to-low-millisecond loop latency. A web browser's audio path is ~20–100 ms — far too slow to cancel live broadband sound acoustically. So our app's real-time-cancellation core will be a physics-accurate SIMULATION (where we control the timing exactly and can prove cancellation), plus an honest, limited real-audio demo. This is the single most important thing to align on — see Decision 1.

1. The core idea

Two sound waves of equal amplitude and opposite phase sum to silence (destructive interference). ANC measures incoming noise, computes its mirror image ("anti-noise"), and plays it so the two cancel at a target point (e.g., your ear).

noise(t) + anti-noise(t) ≈ 0 when anti-noise(t) = −noise(t)

The challenge is that the system doesn't know the noise in advance and the speaker→target path colors and delays the anti-noise — so we need an adaptive filter that learns to pre-compensate. That's FxLMS.

3. The FxLMS algorithm (what we'll implement)

Plain LMS fails here because the anti-noise passes through the secondary path S(z) (speaker + air + mic) before reaching the error mic. FxLMS fixes this by filtering the reference through an estimate Ŝ(z) of that path before the update — hence "filtered-x."

y(n) = wᵀ(n) · x(n) // anti-noise the filter outputs x'(n) = ŝ(n) ∗ x(n) // reference filtered by secondary-path estimate e(n) = d(n) − [ s(n) ∗ y(n) ] // residual at the error mic w(n+1) = w(n) + μ · x'(n) · e(n) // filtered-x LMS update
  • Secondary-path identification: before/while running, we estimate Ŝ(z) (often by injecting low-level white noise and adapting). We'll include this as an explicit step the user can watch.
  • Primary path P(z): noise→error-mic acoustic path; in simulation we define it; the filter implicitly learns to invert it via S.
  • Convergence depends on μ, filter length, and how good Ŝ is.

4. The latency reality (the crux)

For cancellation, anti-noise must arrive within a small fraction of the noise period. The usable upper frequency is roughly set by the loop delay:

f_max ≈ 1 / (2 · total_loop_delay)
SystemLoop delayRealistic ANC ceiling
Dedicated ANC chip (headphones)~10–50 µsup to ~1–2 kHz
Good audio DSP~1 msa few hundred Hz
Browser (Web Audio)~20–100 msonly a few Hz — i.e., not viable for live broadband
Conclusion: we cannot honestly claim real-time acoustic ANC in a browser. We therefore make the cancellation engine a simulation (we control the sample timing precisely, so FxLMS genuinely works and we can show 10–30 dB reduction), and offer real audio only as a narrowband, headphone-based illustration with the limitation stated plainly. This keeps the app scientifically honest — important for your consulting brand.

★ Decision 1 — what "real" means here. Recommended scope:

  2. Real-audio demo (secondary, optional): use the mic to capture ambient noise and play a best-effort anti-noise through headphones for low-frequency / narrowband cases, with an on-screen latency readout and an explicit "illustrative only" notice.

Primary KPI — Noise attenuation

  • Attenuation spectrum — dB reduction vs frequency (shows the low-frequency win and the high-frequency limit clearly).

  • Center/right dashboard: big attenuation-vs-time chart; attenuation spectrum; before/after waveforms; A/B audition buttons.
  • Top status bar: overall dB reduction, convergence state, simulated loop-latency readout + the frequency ceiling it implies.

アクティブノイズキャンセリング (ANC)

マイクで周囲のノイズを検知し、スピーカーから「アンチノイズ」を放射して破壊的干渉でノイズを打ち消す、インタラクティブな Web アプリ — FxLMS 適応アルゴリズムを使用します。

本ドキュメントの目的: ANC の仕組みと、本アプリをどう構築するかの提案を、コーディングに着手する前のレビュー用に説明します。意思決定ポイントは黄色のボックスで明示し、末尾でまとめます。
⚠ 最初にお読みください — 誠実さに関するセクション。 実際の音響 ANC には厳しい因果律／レイテンシ要件があります。アンチノイズは、ノイズとまったく同じ瞬間に、位相を反転した状態でキャンセル地点に到達しなければなりません。これにはマイクロ秒〜低ミリ秒のループレイテンシが要求されます。Web ブラウザのオーディオ経路は約 20〜100 ms — ライブの広帯域音を音響的にキャンセルするにはあまりに遅すぎます。そこで本アプリのリアルタイムキャンセリングのコアは、物理的に正確なシミュレーションとします(タイミングを正確に制御でき、キャンセルを証明できます)。加えて、誠実で限定的な実音声デモを提供します。これは合意すべき最も重要な点です — Decision 1 を参照してください。

1. 核心となるアイデア

振幅が等しく位相が逆の 2 つの音波は、加算すると無音になります(破壊的干渉)。ANC は入ってくるノイズを測定し、その鏡像(「アンチノイズ」)を計算して再生し、対象点(例えば耳の位置)で両者を打ち消します。

noise(t) + anti-noise(t) ≈ 0 when anti-noise(t) = −noise(t)

課題は、システムがノイズを事前に知らないこと、そしてスピーカー→対象点の経路がアンチノイズに色付け(周波数特性の変化)と遅延を加えることです。そのため、あらかじめ補償することを学習する適応フィルタが必要になります。それが FxLMS です。

2. ANC アーキテクチャ

タイプマイク適した用途トレードオフ
フィードフォワード参照 + エラー予測可能／周期的なノイズ(エンジン、ファン)上流側にクリーンな参照が必要
フィードバックエラーのみ広帯域、ヘッドフォン帯域幅が限られ、安定性に敏感
ハイブリッド両方実環境で最良(高級ヘッドフォン)最も複雑

3. FxLMS アルゴリズム(実装するもの)

素の LMS はここでは機能しません。アンチノイズはエラーマイクに届く前に二次経路 S(z)(スピーカー + 空気 + マイク)を通過するためです。FxLMS はこれを、更新の前に参照をその経路の推定値 Ŝ(z) でフィルタリングすることで解決します — これが「フィルタード x(filtered-x)」の由来です。

y(n) = wᵀ(n) · x(n) // anti-noise the filter outputs x'(n) = ŝ(n) ∗ x(n) // reference filtered by secondary-path estimate e(n) = d(n) − [ s(n) ∗ y(n) ] // residual at the error mic w(n+1) = w(n) + μ · x'(n) · e(n) // filtered-x LMS update
  • 二次経路の同定: 実行前または実行中に、Ŝ(z) を推定します(多くの場合、低レベルのホワイトノイズを注入して適応させます)。これをユーザーが観察できる明示的なステップとして組み込みます。
  • 一次経路 P(z): ノイズ→エラーマイクの音響経路。シミュレーションでは我々が定義し、フィルタは S を介してこれを逆転させることを暗黙的に学習します。
  • 収束は μ、フィルタ長、そして Ŝ の精度に依存します。

4. レイテンシの現実(核心)

キャンセルのためには、アンチノイズがノイズ周期のごく一部以内に到達しなければなりません。利用可能な上限周波数は、概ねループ遅延によって決まります。

f_max ≈ 1 / (2 · total_loop_delay)
システムループ遅延現実的な ANC の上限
専用 ANC チップ(ヘッドフォン)約 10〜50 µs最大で約 1〜2 kHz
優れたオーディオ DSP約 1 ms数百 Hz 程度
ブラウザ (Web Audio)約 20〜100 ms数 Hz のみ — つまりライブ広帯域には実用的でない
結論: ブラウザでリアルタイムの音響 ANC を誠実に主張することはできません。そこでキャンセリングエンジンはシミュレーションとします(サンプルタイミングを正確に制御できるため、FxLMS は本当に機能し、10〜30 dB の低減を示せます)。実音声については、狭帯域・ヘッドフォンベースの説明用としてのみ提供し、制約を率直に明示します。これにより、アプリは科学的に誠実なものに保たれます — あなたのコンサルティングブランドにとって重要な点です。

5. 提案するアプリのアーキテクチャ

★ Decision 1 — ここでの「リアル」の意味。 推奨するスコープ:
  1. シミュレーションモード(主軸): ノイズ(エンジン音、ファン、交通、トーン、広帯域)を生成し、P(z) と S(z) をモデル化し、FxLMS を実行して残差が縮小する様子を示します — シミュレーションからレンダリングしたライブの「適用前 vs 適用後」の試聴付き。物理的に正しく、制御可能で、効果を証明できる。
  2. 実音声デモ(補助、任意): マイクで周囲のノイズを取り込み、低周波／狭帯域のケースに対してベストエフォートのアンチノイズをヘッドフォンから再生します。画面上にレイテンシ表示と「あくまで説明用」の明示的な注記を添えます。
シミュレーション優先の枠組みを受け入れていただけますか。それとも、これを変えるような具体的な実キャンセルのユースケース(例えばハードウェアターゲット、ヘッドフォン、固定トーンのノイズ)を想定されていますか。

6. 指標と可視化

主要 KPI — ノイズ減衰量

Attenuation(dB) = 10·log₁₀( E[d²(n)] / E[e²(n)] ) (overall and per-frequency)
  • 減衰スペクトル — 周波数に対する dB 低減量(低周波での効果と高周波での限界を明確に示す)。
  • 時間波形 — ノイズ vs 残差を重ね描きし、収束につれて縮小する様子を示す。
  • 収束曲線 — 時間に対する減衰量。
  • 二次経路推定 Ŝ — 同定ステップが成功する様子を示す。
  • A/B 試聴 — 「ANC オフ」(d を聞く)vs「ANC オン」(e を聞く)— 感情に訴える成果。
  • スペクトログラム／ウォーターフォール(任意)— トーンが消えていく様子を観察。
